<objective>
Close the two high-severity gaps from 03-VERIFICATION.md and 03-REVIEW.md that block Phase 3 completion. The total test surface added by this plan is **exactly 11 new tests** (8 in Task 1, 3 in Task 2), with every test either grep-verifiable or pinned to the real Spring-wired bean via a small set of `internal`-visibility test accessors added alongside the production fix.

**HI-01 — DashboardServer cannot restart.** `DashboardServer.scope` is a `private val` initialised once at class construction (DashboardServer.kt:82). `stop()` cancels it (line 106); a subsequent `start()` then launches the observer on a cancelled scope, producing a silent empty active-agents view. Spring SmartLifecycle contract (and any host that restarts the bean during context refresh) is violated. **Fix:** hold `scope` and `observer` behind `AtomicReference`, recreate both on each `start()`, add `internal fun scopeForTest()` / `internal fun observerForTest()` accessors so the unit test can prove the swap without binding a real port.

**HI-02 — Degraded-mode notice unreachable via Spring wiring.** `KoreAutoConfiguration.inMemoryAuditLog()` unconditionally supplies `InMemoryAuditLog` as the default `AuditLog` bean (KoreAutoConfiguration.kt:62-63). In the zero-config hero-demo Spring path, `DashboardDataService` therefore receives a non-null `InMemoryAuditLog` and `hasStorage()` returns `true`, so the `/kore/fragments/recent-runs` and `/kore/fragments/cost-summary` endpoints render empty tables instead of the UI-SPEC.md degraded copy. **Fix:** add `val isPersistent: Boolean = false` to the `AuditLog` port, override to `true` in `PostgresAuditLogAdapter`, and make `DashboardDataService.hasStorage()` return `auditLog?.isPersistent == true`. Pin the contract to the real Spring-wired bean by adding `internal fun dataServiceForTest()` on DashboardServer and asserting `dashboardServer.dataServiceForTest().hasStorage() shouldBe false` inside a `@SpringBootTest` that lives in kore-spring (bean introspection only — no ktor test host dep in kore-spring).

Purpose: unblock Phase 3 verification gate with minimal, mechanical fixes. Each fix is TDD-ordered (red test first, then code). No scope beyond the two findings. The test split across kore-dashboard and kore-spring follows the checker's Option A recommendation: all HTTP integration assertions live in kore-dashboard (which already has the ktor test host dependency on line 26 of its build.gradle.kts); kore-spring is restricted to bean introspection via the internal accessor, so no new test dependency is added to kore-spring.

- **11 new tests** distributed as: `DashboardDegradedModeTest` (3 unit, kore-dashboard), `DashboardDegradedModeRoutingTest` (2 routing, kore-dashboard), `DashboardDegradedModeSpringTest` (3 spring bean, kore-spring), `DashboardServerRestartTest` (3 unit-level scope swap, kore-dashboard).
