@Configuration(proxyBeanMethods = false)
@ConditionalOnClass(name = ["dev.unityinflow.kore.kafka.KafkaEventBus"])
@ConditionalOnProperty(prefix = "kore.event-bus", name = ["type"], havingValue = "kafka")
class KafkaEventBusAutoConfiguration {
    @Bean(destroyMethod = "close")

        @Qualifier("koreEventBusScope") scope: CoroutineScope,

KafkaEventBus public factory: `operator fun invoke(config: KafkaEventBusConfig, scope: CoroutineScope): KafkaEventBus`

       import dev.unityinflow.kore.core.internal.InProcessEventBus

               contextRunner
                   .withPropertyValues(

                       "kore.event-bus.kafka.bootstrap-servers=localhost:9092",
                       "kore.event-bus.kafka.topic=kore-agent-events",

                   }
           }

           @Test
           fun `type=in-process keeps InProcessEventBus default`() {
               contextRunner
                   .withPropertyValues("kore.event-bus.type=in-process")
                   .run { ctx ->
                       ctx.getBean(EventBus::class.java).shouldBeInstanceOf<InProcessEventBus>()
                   }
           }

           @Test
           fun `type property absent keeps InProcessEventBus default`() {
               contextRunner.run { ctx ->
                   ctx.getBean(EventBus::class.java).shouldBeInstanceOf<InProcessEventBus>()
               }
           }
       }

Alternatively, the auto-configuration could create the bean lazily (`@Lazy`) — but the simpler pattern is to run context tests with bean-definition inspection, and run a separate integration test against a Testcontainers broker when asserting actual bean resolution.

       **Better approach (USE THIS):** Make the `@Bean` method lazy by having `KafkaEventBus.invoke(config, scope)` defer actual producer/consumer construction until first use — but that's a refactor of plan 04-02. Simpler: the Spring context test declares a `@Configuration` that provides a mock `KafkaProducer` + `KafkaConsumer` via `ContextRunner.withUserConfiguration(MockKafkaTestConfig::class.java)`, and the `KafkaEventBusAutoConfiguration` uses an internal `createForTest` factory when those beans are present. This is too invasive for v0.0.1.

       **Cleanest pragmatic approach (ADOPT):** The Spring context test asserts bean definitions exist (as above), and a separate integration test in kore-kafka (plan 04-02 task 1 `KafkaEventBusIntegrationTest`) already covers actual bean resolution against a real broker. Keep the Spring test at the "declaration exists" level.
